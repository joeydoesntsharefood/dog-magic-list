# US03.3: Regras Avançadas de Commander e Multi-Add UI

**Descrição:** Implementar a seleção obrigatória de Comandante, validação de Identidade de Cor e sistema de adição rápida de múltiplas cópias (exeto Commander).

## Critérios de Aceite (AC):

### 1. Novo Step: Seleção de Comandante (Exclusivo Commander)
*   Antes da fase de construção, o usuário deve pesquisar e selecionar uma carta para ser o Comandante.
*   **Regra de Filtro:** Apenas cartas que sejam `Creature` E `Legendary` (ou possuam a regra específica de comandante no Scryfall).
*   O sistema deve extrair a **Identidade de Cor** (Color Identity) do comandante selecionado.

### 2. Validação de Identidade de Cor
*   Durante a construção do deck, qualquer carta adicionada que possua cores fora da identidade do comandante deve ser marcada como `RULE_VIOLATION: COLOR_IDENTITY_OUT_OF_BOUNDS`.

### 3. Sistema de Multi-Add (UX)
*   Na UI de construção, adicionar controles de quantidade `[-] 1 [+]` antes do botão de commitar.
*   **Regra Commander:** O controle de quantidade deve vir desabilitado ou travado em `1` por padrão para evitar quebras de Singleton (exceto terrenos básicos).
*   Se o usuário tentar adicionar uma carta que já existe no deck em formatos Singleton, o sistema deve impedir a ação com a mensagem `ACTION_DENIED: SINGLETON_CONSTRAINT`.

### 4. Refinamento de Singleton
*   A violação de Singleton deve ser checada no momento da tentativa de adição, impedindo o "commit" para o deck se for uma duplicata proibida.

## Estratégia Técnica:
*   **Backend:** Adicionar `color_identity` no retorno da `CardVersion`.
*   **Frontend:** Novo estado `commander` no `deckProfile`.

## Metodologia TDI:
*   **Teste de UI:** Validar se o step de comandante aparece apenas quando o formato for COMMANDER.
*   **Teste de Lógica:** Garantir que cartas fora da cor do comandante disparem o erro visual.
