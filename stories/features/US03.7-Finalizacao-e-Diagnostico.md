# US03.7: Finalização de Deck - Diagnóstico vs Persistência

**Descrição:** Implementar a etapa final de decisão no Wizard, permitindo ao usuário escolher entre realizar um diagnóstico estatístico completo do deck ou salvá-lo diretamente no banco de dados.

## Critérios de Aceite (AC):

### 1. Novo Step: DECISION_PROMPT (Step 6)
*   Após o ajuste fino de edições, apresentar dois caminhos claros:
    *   **Botão [RUN_DIAGNOSTICS]:** Encaminha para a tela de análise técnica (Pilares, Curva, Draw).
    *   **Botão [COMMIT_TO_GRIMOIRE]:** Salva o deck no SQLite e retorna à tela inicial.

### 2. Tela de Diagnóstico (D2D Analysis)
*   Exibir dashboard minimalista com:
    *   **Mana Curve:** Histograma de densidade de custo.
    *   **Pillar Status:** Barra de progresso para Interação, Draw e Ramp (Baseado nos alvos da doc).
    *   **Draw Sim:** Probabilidade de mão ideal (Hipergeométrica).
*   Permitir salvar o deck a partir desta tela também.

### 3. Persistência (SQLite Integration)
*   O deck deve ser salvo com: Nome, Formato, Lista de Cartas (ID da versão escolhida), Preço Total e Diagnóstico Resumido.

## Estratégia Técnica:
*   **Backend:** Endpoint `POST /decks` para salvar a estrutura completa.
*   **Frontend:** Componentes de visualização de dados (BarChart minimalista ou apenas ASCII/CSS Bars).

## Metodologia TDI:
*   **Teste de Integração:** Validar se o salvamento do deck cria os registros corretos na tabela de listas e em uma nova tabela `deck_contents`.
