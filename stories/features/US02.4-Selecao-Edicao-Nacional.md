# US02.4: Resolução de Ambiguidade e Seleção de Edição (LigaMagic)

**Descrição:** Implementar um fluxo de seleção de edição quando a busca na LigaMagic retornar múltiplas cartas (MTG-Singles) ou variações da mesma carta, garantindo que os dados de mercado correspondam à escolha do usuário.

## Critérios de Aceite (AC):
1.  **Identificação de Ambiguidade:**
    *   O scraper deve capturar todos os elementos `.mtg-single` na página de resultados.
    *   Se houver mais de um resultado ou se a página for uma lista de variações (ex: Island), o backend deve retornar a lista de opções para o frontend.
2.  **UI de Seleção de Edição:**
    *   O frontend deve exibir um modal ou lista de "Edições Encontradas na LigaMagic".
    *   Cada opção deve exibir o nome da coleção e, se possível, a imagem da edição.
3.  **Navegação Direta:**
    *   Ao usuário selecionar uma edição, o sistema deve disparar o scraper diretamente para a URL específica daquela edição (ex: `view=cards/card&ed=...`).
4.  **Persistência Precisa:**
    *   Os dados de mercado (preço médio e ofertas) devem ser salvos no DB vinculados àquela edição específica.

## Estratégia Técnica:
*   **Backend:** Novo endpoint `GET /offers/options` para listar as edições disponíveis antes de iniciar o scraping de ofertas.
*   **Scraper:** Refinar a captura de `.mtg-single` para extrair ID de edição (`ed`) e número (`num`).

## Metodologia TDI:
*   **Teste Unitário:** Validar a extração de múltiplos `.mtg-single` de um HTML de busca ambíguo.
*   **Teste de Integração:** Simular o fluxo: Busca -> Recebe Opções -> Seleciona Opção -> Recebe Ofertas.
